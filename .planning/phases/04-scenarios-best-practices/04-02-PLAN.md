**Slide 1: Mental model shift 1 (BEST-01)**

Title: `## Tänk outcomes, inte steg`

Body: Show a ❌/✅ do/don't pair to illustrate the shift from implementation-thinking to outcome-thinking:

❌ (bold label "Steg-för-steg:"):
> "Create a file called AuthService.ts, add a login method that accepts email and password, hash with bcrypt, create a JWT token..."

✅ (bold label "Outcome:"):
> "Användare ska kunna logga in med email och lösenord"

Below the pair: `GSD härleder stegen från målet. Du definierar vad som är klart — inte hur.`

**Slide 2: Mental model shift 2 (BEST-01)**

Title: `## Du bestämmer, AI utför`

Body: Two-part division showing the role split:

**Du:**
- Vad ska byggas
- Arkitekturbeslut
- Kvalitetskrav

**AI:**
- Research och analys
- Planering och implementation
- Verification execution

Below: `Utvecklaren blir arkitekt och reviewer — inte typist.`

**Slide 3: Prompt crafting (BEST-02, D-10)**

Title: `## Prompt crafting — var specifik`

Body: Show 3-4 concrete good/bad prompt pairs. Use ❌/✅ format or a compact layout:

❌ `"Fix the bug"`
✅ `"The login form returns 401 when valid credentials are submitted"`

❌ `"Add a button"`
✅ `"Users should be able to export their data as CSV from the settings page"`

❌ `"Refactor this"`
✅ `"Refactor the auth module — it mixes JWT creation, validation, and user lookup in one file"`

Below the examples: `Kontext + mål + constraints = bättre resultat.`

Key design note: These must be CONCRETE examples per D-10, not abstract principles. The good prompts include context, goal, and constraints. The bad prompts are vague.

**Slide 4: Anti-patterns (BEST-04)**

Title: `## Tre vanliga misstag`

Body: Present three anti-patterns, each as a bold label with a one-line description and a one-line "instead" guidance:

**🚫 Vaga prompts** — agenten gissar fel, du gör om arbetet
→ Var specifik: mål + kontext + constraints

**🚫 Skippa verification** — "3/3 tasks done!" men inget fungerar
→ Kör alltid `/gsd-verify-phase` — task completion ≠ goal achievement

**🚫 Micro-management** — du dikterar varje fil och rad
→ Beskriv outcome, låt GSD planera vägen dit

The verification anti-pattern MUST reference the concept from Section 5 ("task completion ≠ goal achievement") — this connects back to the verification philosophy climax.

**Slide 5: Escape hatches (BEST-03, D-09)**

Title: `## När du INTE ska använda agents`

Body: 5 concrete situations where agents should NOT be used, each with a 🚫 emoji and a short explanation. This slide must carry EQUAL WEIGHT to the anti-patterns slide — it gets its own full slide, not a footnote:

🚫 **Arkitekturbeslut** — agenten optimerar lokalt, du optimerar globalt

🚫 **Security-kritisk kod** — alltid manuell review oavsett

🚫 **Teamets lärande** — om syftet är att lära, inte leverera

🚫 **Regelverk & compliance** — agenten förstår inte juridiskt ansvar

🚫 **Det du inte kan reviewera** — delegera inte det du inte kan bedöma

Below the list, a blockquote: `> GSD gör dig effektivare — men det är fortfarande du som är ansvarig.`

This builds credibility with skeptics. The tone is professional honesty, not weakness.
